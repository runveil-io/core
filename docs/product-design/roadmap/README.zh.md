# Roadmap

## 目的

本节说明 Veil 的分阶段落地顺序。它是一个面向后续实施的排序文档。

除非其他章节明确说明已经实现，否则本节内容都应视为路线图范围，而不是当前运行时现状。

## 规划原则

每个阶段都应该让系统比之前更可运行、更可验证。新能力不能跑在协议清晰度、安全边界和运行时可见性前面。

当路线图被拆成实施任务时，应以 [执行规则](./execution-rules.zh.md) 作为硬依赖顺序和阶段门槛的参考。
当判断哪些能力必须保留在主线上时，应以 [产品愿景](../../design-governance/product-vision.zh.md) 作为产品约束来源。

## 北极星

Veil 的目标运维体验是：

1. 告诉 Claw 接入网络
2. 告诉 Claw 开始出售可用推理容量
3. 让系统以尽量少的人工干预持续保持节点在线、定价可控、路由可用、见证记录完整且可恢复

## 规划主线

- `Autopilot`：减少人工节点生命周期操作
- `Marketplace`：把容量变成可出售的网络供给
- `Settlement`：从见证记录和报价单位走向可结算、兼容 crypto 的账务
- `Privacy`：在市场闭环增长的同时继续保持无账户接入和隐私保护式路由

## Phase 1：可靠运行时基线

- 把本地 Consumer 网关稳定为默认的无账户接入入口
- 稳定 Relay 到 Provider 的请求转发
- 让真实 Consumer 到 Provider 路径上的见证记录、预算与错误链路可用

## Phase 2：发现与多 Relay 可达性

- 完善 Bootstrap 与 Relay 发现
- 支持更好的 Relay 选择
- 提升 Provider 的多 Relay 可达性
- 让 Consumer 更容易发现并使用可用的 Provider 供给

## Phase 3：引导式运维自动化

- 强化运行时限制与连接行为
- 加强 Relay 与 Provider 的健康管理
- 收紧秘密处理和运维流程
- 让 Provider 作为经济角色运行时更安全、更可预测
- 提供第一版由 Claw 托管的接入和启动流程，避免运维者手工拼装所有运行步骤

## Phase 4：自治市场控制

- 改进重放保护和请求契约
- 加强用量与见证记录证据
- 提升协议版本和兼容策略
- 引入按策略驱动的容量发布和定价控制
- 让 Claw 在尽量少人工干预下保持卖方策略、健康状态和故障恢复的一致性

## Phase 5：基于见证记录的结算

- 保持治理账务与推理账务协同但分离
- 完善贡献账务与导出流程
- 让定价与基于见证记录的结算接口可用
- 明确区分预算或报价使用的计价单位，与支付轨使用的结算资产
- 为支付轨道以及 Provider 或 Relay 收益分发系统预留更清晰接口
- 让 crypto 原生结算继续保留在主路径上，而不是变成可有可无的附属项
- 在不把两者混成一本账的前提下对齐产品市场闭环与开源构建循环

## Phase 6：低人工干预市场运营

- 让 Claw 以接近自治的方式运行 Provider 节点
- 自动完成调价、暂停与恢复，以及卖方风险包络控制
- 把人工配置尽量收敛到运维者意图、策略护栏和凭证授权

## Phase 7：可选的高级隐私配置

- 只在市场闭环具备运行能力之后，才探索更强的隐私配置
- 明确区分标准市场模式和未来可能的强匿名模式
- 避免让匿名性表述跑在实际系统保证前面
