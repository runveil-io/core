# 产品愿景

## 目的

本页用于明确 Veil 的产品愿景，并将其作为后续架构与路线图规划的统一方向。
它的职责是确保架构和路线图在演进过程中持续服务于 Veil 的市场、隐私和自动化目标。

## 愿景

Veil 是一个由 agent 驱动的 AI 容量市场网络。  
它让 Consumer 获得低摩擦、无账户绑定的 AI 接入，让 Provider 和 Relay 形成可持续的供给与撮合关系，并在隐私保护与可结算性之间建立长期可演进的系统能力。

## 核心结果

- `Access`：默认提供本地 OpenAI 兼容接入，降低用户与工具接入成本
- `Privacy`：通过隐私保护式路由减少身份与请求可见性耦合
- `Market`：让 Provider 与 Relay 的供给、撮合与收益成为真实工作流
- `Automation`：以 agent 自动化覆盖接入、发布、运行与恢复过程
- `Settlement`：以见证记录和定价能力支撑可支付、可结算的演进路径

## 产品原则

- **市场优先**：Veil 不是单纯传输层，而是连接供需关系的网络市场
- **低摩擦**：接入和运维体验应优于直接拼装上游 API
- **隐私内建**：隐私是默认产品属性，而不是可有可无的附加层
- **自动化内建**：agent 能力是产品表面的一部分，而不是仅限内部工具
- **持续可结算**：账务与结算能力可以分阶段实现，但必须保持主线地位

## 架构约束

后续任何架构设计都应保留这些约束：

1. 本地 OpenAI 兼容接入仍然是默认集成面
2. Provider 容量发布必须成为真实的卖方工作流，而不只是背景概念
3. Relay 必须继续作为显式的市场角色，承担路由和见证记录职责
4. 见证记录和定价必须能继续喂给未来结算
5. Claw 或等价自动化层必须进入节点接入和运营链路
6. 隐私口径可以收紧，但架构仍应继续优化可见性拆分和身份去耦合

## 路线图约束

因此路线图必须同时维持五条结果主线：

- `Access`：保持本地网关和客户端兼容能力
- `Privacy`：把隐私保护式路由与降低身份耦合保持在主线路线图中
- `Market`：让 Provider 和 Relay 的经济角色变成现实
- `Automation`：减少接入、发布和恢复所需的人工操作
- `Settlement`：把基于见证记录的账务推进到可支付工作流

如果某一轮路线图只提升了运行时稳定性，却削弱了上述主线结果，就应视为偏离产品愿景。

## 适用场景

- 评审路线图变更
- 引入新的核心模块
- 为了简化架构而可能弱化市场或自动化目标时
- 判断一个能力应属于产品核心还是仅属于工具链
