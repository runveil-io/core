# 文档治理规则

## 目的

本页定义 Veil 文档的归类、命名、评审与维护规则，确保产品设计与技术设计长期保持一致。

## 规范分类

### `docs/product-design/`

用于产品侧意图与方向：

- 产品范围与定位
- 信任与隐私口径
- 治理与经济边界
- 路线图排序

不应放入协议细节、配置键说明或模块合同。

### `docs/technical-design/`

用于实现侧系统设计：

- 系统模型与运行职责
- 架构平面与有界上下文
- 模块合同
- 协议契约
- 运行时配置面

不应放入产品叙事或路线图政策说明。

### `docs/design-governance/`

用于跨产品与技术的对齐治理：

- 产品愿景
- 追踪矩阵
- 文档治理规则

不应放入教程、运行手册或端点使用示例。

### `docs/clients/`、`docs/installation/`、`docs/manual/`、`docs/operations/`、`docs/glossary/`

这些目录保留为接入、安装、运行、运维和术语等功能域。

## 命名规则

- 目录与文件统一使用 kebab-case
- 每个目录必须包含 `README.md` 与 `README.zh.md`
- 中英文配对文档必须保持同名基名：
  - `name.md`
  - `name.zh.md`
- 避免使用含糊命名，如 `notes.md`、`misc.md`、`draft-v2.md`

## 文档结构规则

每个分区 README 建议包含：

- 目的
- 本分区应包含什么
- 本分区不应包含什么
- 下一步阅读

规范型文档（愿景、矩阵、治理规则）还应说明如何用于评审。

## 链接规则

- 每篇文档应向上链接到所在分区入口
- 每篇文档应至少链接一个相关邻接分区
- 产品设计文档应至少链接一个治理锚点
- 技术设计文档应至少链接一个产品或治理依据

## 变更管理规则

新增或迁移文档时：

1. 在 PR 中说明归属理由
2. 更新父级 `README` 导航
3. 保持受影响文档的中英文对齐
4. 合并前执行链接完整性检查

废弃文档时：

- 在可行情况下先保留一个评审周期的跳转占位页
- 确认引用全部更新后再删除

## 评审清单

- 该文档是否按“目的”放到了正确目录？
- 命名是否合规，中英文是否配对完整？
- 分区与根入口导航是否已更新？
- 文案是否与产品愿景和追踪规则保持一致？
- 是否与现有规范文档产生重复来源？

## 快速归档决策树

- 首次安装 -> `docs/installation/`
- 角色化操作流程 -> `docs/manual/`
- 运行可观测与发布门槛 -> `docs/operations/`
- 客户端接入示例 -> `docs/clients/`
- 产品意图、隐私口径、经济边界、路线图 -> `docs/product-design/`
- 架构、模块、协议、配置 -> `docs/technical-design/`
- 跨域对齐约束与追踪治理 -> `docs/design-governance/`
- 统一术语定义 -> `docs/glossary/`
