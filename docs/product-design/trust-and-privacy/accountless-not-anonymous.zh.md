# 无账户接入，不等于匿名

## 目的

本页解释为什么 Veil 的公开表述是 `无账户接入` 和 `隐私保护式路由`，而不是承诺匿名。

## 简短结论

Veil 的价值在于降低账户直连耦合，并把可见性拆分到不同角色之间。这很重要，但它不等于对所有运维者、上游提供方或外部观察者都成立的匿名性证明。

这次表述收紧不是回退 Veil 的市场、自动化或隐私目标，而是把公开承诺修正到精确边界上。

## Veil 里的 `无账户接入` 是什么意思

- 本地客户端通过一个本地网关接入 Veil，而不是在每个工具里分别嵌入上游账户流程
- Relay 可以撮合流量，但不成为模型执行端点
- Provider 可以执行请求，但不必拿到完整的 Consumer 本地环境

`无账户接入` 说的是正常使用路径不再要求每个客户端直接绑定独立上游账户流程。它不是在说所有身份或关联信号都会消失。

## 为什么 Veil 不承诺 `匿名`

强匿名要求 Veil 防御的不只是提示词泄露。

- Relay 仍然会看到路由和见证记录元数据
- Provider 仍然会看到明文执行载荷
- 上游模型提供方仍可能看到请求时序、模型使用和账户级行为
- 流量分析、节点串谋和端点被攻破，都不在当前设计的保证范围内

因此，公开文档不应把 Veil 描述成“保证匿名”“无法追踪”或“不会被运维者关联分析”。

## 为什么移除了 `Relay 看谁，Provider 看什么`

这句话可以作为简化解释，但作为公开安全表述过于压缩。

它说对的部分：

- Relay 路由请求时不需要提示词明文
- Provider 处在执行边界之内

它隐藏的部分：

- Relay 仍然可以看到连接级和路由级元数据
- Provider 仍可能从载荷结构、工作负载形态或重复任务行为推断上下文
- 这句话并不能证明在节点串谋或上游可见性存在时还能成立匿名性

现在的表述更精确：Relay 看到的是路由和见证记录元数据，而不是提示词明文；Provider 看到的是明文执行载荷，但不应接收不必要的 Consumer 本地上下文。

## 为什么把 `relay.runveil.io` 从规范入口移除了

单一硬编码 Relay 地址可以作为示例，但不应该定义产品架构。

Veil 现在正式文档写的是：

- 可配置的 Relay 端点
- 基于 Bootstrap 的 Relay 发现
- 多 Relay 运行作为系统方向的一部分

如果在规范入口和贡献文档里持续保留一个固定公网 Relay 地址，会带来错误预期：

- 看起来像 Veil 永远是单运维者系统
- 会削弱 Bootstrap 和发现机制在架构里的作用
- 会把某个部署细节误写成协议假设

如果 Veil 以后长期运行官方公共 Relay，它应该出现在运维或上手示例中，或者被说明为默认入口，而不是核心信任模型本身。重点是避免把默认入口固化成协议假设。

## 为什么移除了 `clawd build`

`clawd build` 不是“完全没用”，它更像是一种内部工作流设想。它之所以从规范入口移除，是因为它不是仓库今天已经支持的公开接口。

公开文档只应该描述这些内容：

- 外部贡献者今天就能实际使用
- 能在仓库里复现
- 与当前命令面一致

当前仓库真正暴露的是：

- `veil` 作为运行时 CLI
- `npm test` 等开发命令
- `desired/*.yaml`、失败测试和 TODO 作为贡献来源

如果在规范文档里写一个仓库外部并不存在的 agent 命令，只会让贡献者误以为这是官方支持能力。被移除的是“不受支持的命令写法”，不是“Claw 自动化”这个产品目标本身。

产品愿景仍然要求 Claw 最终回到正式产品面，成为接入网络、卖量和运行节点的自动化入口。当前文档已经把这条主线收敛到了 `Claw Autopilot` 模块和路线图里。

## 公开表述规则

建议使用：

- 无账户接入
- 隐私保护式路由
- 通过角色拆分实现可见性分离

避免使用：

- 默认匿名
- 无身份
- 无跟踪
- 不可关联
