# 设计治理

## 目的

本节用于定义产品设计与技术设计对齐时的治理基线。

规范约束来源见 [产品愿景](./product-vision.zh.md)。

## 适用场景

- 你想理解 Veil 的用户侧和运维侧主张
- 你需要知道为什么 Consumer、Provider、Relay 都有存在意义
- 你想在看实现细节前先看清产品闭环

## 产品主张

Veil 把 AI 接入组织成一条可路由的网络路径：

- Consumer 使用统一的本地 OpenAI 兼容网关
- Provider 提供闲置 AI 容量或 API 配额
- Relay 负责撮合流量和记录见证记录，但不成为执行端点

产品目标是降低用户工具与单一上游账户之间的强绑定，同时保留执行过程的可审计性。

## 意图锚点

最早公开版 Veil 的产品承诺范围大于当前运行时边界。但这些初衷仍然约束今天的规划：

- Veil 目标上是一个市场，而不只是路由层
- Claw 目标上应进入产品表面，而不是永远停留在外部工具位置
- 结算和卖方经济结果属于核心产出，而不是可有可无的附属项

如果后续要弱化这些主线，应先回看 [产品愿景](./product-vision.zh.md)。

## 北极星体验

Veil 的终局目标是把人工节点运维压到最低。理想体验是：运维者只需要告诉 Claw 接入网络并开始出售可用容量，Veil 就自动处理后续运行链路：

- 接入或发现可达的 Relay 基础设施
- 注册 Provider 能力和容量
- 应用定价与风险策略
- 接收并执行可路由需求
- 记录见证记录并驱动结算
- 在条件变化时自动暂停、恢复或调价

## 核心角色

- `Consumer`：希望通过一个本地接口获得可用的 AI 接入
- `Provider`：希望把闲置容量暴露成可路由供给
- `Relay`：希望运行路由与见证记录基础设施
- `Claw`：希望代表运维者自动完成节点接入、运营、定价与恢复
- `Contributor`：希望通过开源工作推动协议和工具演进

## 产品承诺

- 本地 OpenAI 兼容接入
- 通过 Veil 路径实现无账户接入，而不是让每个客户端逐个直连上游
- 可复用的 Provider 侧供给
- 隐私保护式的角色拆分
- 以见证记录为基础的账务边界
- 朝着低人工干预节点运营和自动出售可路由容量演进

## 主要场景

- 用户希望继续使用现有工具获取 AI 接入，而不是给每个工具都绑定独立上游账户
- Provider 希望把闲置配额或容量转换成可用网络供给
- Relay 运维者希望撮合流量，同时不需要看到提示词明文
- 运维者希望让 Claw 自动接入网络、发布容量，并在尽量少人工干预下持续卖量
- Contributor 希望改进系统，并保留可审计的贡献轨迹

## 价值流

```text
Consumer 需求 -> Relay 路由 -> Provider 执行 -> 见证记录 -> 定价 / 结算
                           \-> 开源贡献 -> RBOB 记账
```

定价与结算属于产品方向接口。当前运行时已经提供见证记录和账务边界，但还没有暴露完整的支付轨道或收益分发接口。

## 当前边界与目标边界

- 当前运行时：仍以显式 CLI 启动、手工配置、静态 Relay 默认值和人工管理 Provider 生命周期为主
- 目标运行时：由 Claw 托管接入流程、自动发现 Relay、按策略定价、自动发布容量，并驱动基于见证记录的结算工作流

## 产品红线

- 正常使用 Veil 时不应要求自定义客户端协议
- 不应对匿名性做不加限定的承诺
- 不应把绕过上游规则作为公开产品能力
- 不应把推理见证记录和贡献账务混成一条数据路径
